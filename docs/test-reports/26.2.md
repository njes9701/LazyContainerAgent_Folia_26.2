## 測試 2 — 資料完整性 3-boot round-trip（決定性）
程序（`run-test.sh`）：BOOT1 放箱+存 → BOOT2 磁碟 lazy 載入 + 未碰即 raw 回寫 → BOOT3 讀回比對。
- 放入：`chest{Items:[diamond×42, diamond_sword{components:{damage:10}}, netherite_ingot×7]}`
- BOOT3 讀回：
```
[{Slot:0b, count:42, id:"minecraft:diamond"},
 {Slot:5b, count:1, components:{"minecraft:damage":10}, id:"minecraft:diamond_sword"},
 {Slot:9b, count:7, id:"minecraft:netherite_ingot"}]
```
- **物品 id、count、data-component(`damage:10`)全保留。**
- 各 boot 計數：
  | boot | stash | rawSave | ensure | eagerLoad | shadowMismatch | benignReorder |
  |---|---|---|---|---|---|---|
  | 1（放箱+存）| 5 | 7 | 0 | 0 | **0** | 0 |
  | 2（lazy 載入+raw 回寫）| 1 | 1 | 0 | 0 | **0** | 0 |
  | 3（讀回）| 1 | 2 | 0 | 0 | **0** | 0 |
- `stash>0`＝lazy 載入生效；`rawSave>0`＝raw 回寫生效；`shadowMismatch=0`＝raw 回寫與 vanilla encode 逐位元組相同 → **✅ PASS**。

## 覆蓋與限制
- 已驗：注入正確性、跨重啟資料完整性（含 component）、真實世界 shadow byte-identical。
- 未在本報告涵蓋：26.2 的效能 benchmark（1.21.11 的 spark 數據見 [`../spark/SUMMARY.md`](../spark/SUMMARY.md)；26.2 機制相同，效益等價，待正式節點 spark 量測）。
- ensure（decode-on-access）路徑在 headless 測試未觸發（無玩家/漏斗互動，`ensure=0`）；其正確性由 round-trip 的讀回比對與 shadow 間接覆蓋。

## 上線建議
照原 v1.0 流程：第一次以 `-Dlazycontainer.shadow=true -Dlazycontainer.verbose=true` 開機，跑數天確認 `shadowMismatch=0` 後再移除 shadow 取效能。回退＝移除 `-javaagent` 旗標重啟（磁碟格式從未改變，無資料遷移）。
